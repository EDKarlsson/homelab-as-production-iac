Cross-cutting gotchas encountered across the homelab infrastructure stack. Each entry is a confirmed issue with a known fix or workaround.

## Terraform / Proxmox Provider

| Area | Gotcha | Fix |
|------|--------|-----|
| bpg/proxmox | `user_account` and `user_data_file_id` are mutually exclusive in v0.95.0 | Use one or the other, not both |
| bpg/proxmox | SSH username in provider must be the OS user (`root`), not the PVE API user | Set `ssh { username = "root" }` |
| bpg/proxmox | GNOME Keyring SSH agent (`/run/user/1000/keyring/ssh`) incompatible with Go SSH lib -- provider fails silently | Pass `private_key` from 1Password directly to the provider |
| bpg/proxmox | `agent { enabled = true }` makes Terraform wait for qemu-guest-agent; missing agent = plan hangs for minutes | Include `qemu-guest-agent` in cloud-config packages AND `systemctl start qemu-guest-agent` in runcmd (Ubuntu 24.04 installs but does not auto-start) |
| bpg/proxmox | Auto-creates cloud-init drive (ide2) from `initialization` block; `user_data_file_id` replaces only user-data | Network-config is generated from `ip_config`/`dns` blocks |
| Proxmox VM | VM `name` attribute must be DNS-valid (hyphens); Terraform resource names use underscores | Use hyphens in `name`, underscores in resource identifiers |
| Proxmox storage | `download_file` needs `import` content type -- only `Proxmox_NAS` has it, not `local-lvm` | Stage downloads to NAS, then use `import_from` to copy |
| Proxmox storage | `import_from` copies at create time only (NAS -> local-lvm), not ongoing sync | Re-create resource to pick up image changes |
| Terraform HCL | `keys()` returns map keys in lexicographic order (guaranteed, unlike most languages) | Rely on this ordering when using `index(keys(map), key)` |
| Terraform HCL | `index()` takes a list, not a map | Use `index(keys(map), key)` to get numeric index from a map |
| Terraform destroy | `terraform destroy -refresh=false` destroys ALL state including VM template | Use `-target=module.k3s` to spare resources outside the module |
| bpg/proxmox env | `PROXMOX_VE_API_TOKEN`, `PROXMOX_VE_USERNAME`, `PROXMOX_VE_PASSWORD` env vars silently override the `provider "proxmox" {}` block | Never set `PROXMOX_VE_*` env vars; verify with `env \| grep PROXMOX_VE` (see [environment-variables.md](./environment-variables.md)) |
| bpg/proxmox | Changing `user_data_file_id` forces VM replacement since cloud-init only runs on first boot | Add `lifecycle { ignore_changes = [initialization] }` to the VM resource to prevent replacement on cloud-init changes |
| bpg/proxmox cloud-init snippets | `proxmox_virtual_environment_file` with `source_raw.data` containing sensitive values (tokens, passwords) is never stored in Terraform state -- every `plan` shows it as changed, forcing `proxmox_virtual_environment_file` replacement, which cascades to VM replacement via `user_data_file_id # forces replacement` | The fix is `lifecycle { ignore_changes = [initialization] }` on the **VM resource** (not the file resource); this prevents the cascade even when the snippet file is replaced. Confirmed in the k3s module where cloud-init snippets embed the cluster token |

## Proxmox LXC Containers

| Area | Gotcha | Fix |
|------|--------|-----|
| LXC + Docker | Docker-in-LXC requires `unprivileged = false`, `nesting = true`, AND `keyctl = true` | All three settings are needed; missing any one causes Docker daemon or storage driver failures |
| LXC provisioning | LXC containers do NOT support cloud-init (no `user_data_file_id`, no `packages`, no `runcmd`) | Use `null_resource` with SSH provisioners for software installation and configuration |
| LXC provisioning | Inline provisioners on the container resource cause the LXC to be tainted on provisioner failure | Separate provisioning into `null_resource` -- allows re-provisioning via `terraform taint` without recreating the container |
| LXC user_account | The `password` field is required by the provider even when SSH key auth is the primary method | Set a placeholder password; SSH key auth is still used for ongoing access |
| LXC templates | `download_file` for LXC templates requires `content_type = "vztmpl"` (not `iso` or `import`) | Use `local` storage (supports vztmpl by default); `Proxmox_NAS` may not have vztmpl enabled |
| LXC templates | `download_file` errors with "File already exists" if template was pre-downloaded via `pveam` | Add `overwrite_unmanaged = true` to adopt pre-existing files into Terraform management |
| LXC networking | Same VLAN double-tagging issue as VMs: do not set `vlan_id` if vmbr0 is already on an access port | Omit `vlan_id` from `network_interface` to inherit VLAN from the bridge |
| Keepalived VRRP | Multicast VRRP (default) is unreliable in Proxmox virtualized environments | Always use `unicast_src_ip` + `unicast_peer` for deterministic VRRP in LXC containers |
| Keepalived | `vrrp_script weight` must exceed the priority gap between MASTER and BACKUP for failover to trigger | If MASTER=100 and BACKUP=90 (gap=10), use `weight -20` so failed MASTER drops to 80 (below BACKUP) |
| LXC boot order | Infrastructure LXC containers (secrets, DNS) must boot before dependent VMs | Use `startup { order = "1" }` with `up_delay` to ensure services are ready before next tier boots |
| LXC auth | Privileged LXC feature flags (`nesting`, `keyctl`, `fuse`) require `root@pam` username/password auth -- API tokens fail the string equality check `$authuser ne 'root@pam'` even with `privsep=0` | Use `username = "root@pam"` + `password` from 1Password, not API tokens |
| LXC + Docker | Docker Compose services in LXC may fail with AppArmor denials | Add `security_opt: [apparmor:unconfined]` to the service in `docker-compose.yml` |
| LXC + Docker | `systemctl disable apparmor` is not enough for Docker-in-LXC -- Docker checks for the kernel module at startup, not the service state | Remove the `apparmor` package entirely (`apt-get remove --purge apparmor`). The host's AppArmor already confines the LXC container; the in-guest copy is redundant |
| LXC + Docker | `gpg --dearmor` hangs on re-provision when Docker is already installed (GPG keyring locked or stdin blocked) | Guard Docker installation with `command -v docker` and skip the entire install block if Docker is already present |
| LXC + Docker | Docker Compose `localhost` does not cross container boundaries -- services cannot reach each other via `localhost:<port>` | Use Docker Compose service names as hostnames (e.g., `connect-api:11220` instead of `localhost:11220`) |
| LXC + Docker | 1Password Connect credentials file with `chmod 600` is unreadable by `opuser` (UID 999) inside the container | Use `chmod 644` for the credentials file so the non-root container user can read it |
| LXC + Docker networking | Docker containers running inside an LXC inherit the LXC's bridge interface (e.g., `docker0`), NOT the Proxmox host's network. The Tailscale resolver (`100.100.100.100`) is only reachable from the Proxmox host, not from inside the LXC bridge network. Docker containers within the LXC that rely on the LXC's `/etc/resolv.conf` (which may point at `100.100.100.100`) will fail DNS resolution. | Configure a custom DNS server in `/etc/docker/daemon.json` inside the LXC: `{ "dns": ["8.8.8.8", "8.8.4.4"] }`, then `systemctl restart docker`. This overrides the inherited resolv.conf for all Docker containers inside the LXC. |

## Proxmox Cluster Management

| Area | Gotcha | Fix |
|------|--------|-----|
| `pvecm add` TLS | `pvecm add <host>` performs a TLS certificate hostname check. The PVE API cert CN is `<node>.lan`, so using a bare hostname or IP may fail. | Use `--fingerprint $(openssl s_client -connect <node>:8006 </dev/null 2>/dev/null \| openssl x509 -noout -fingerprint -sha256 \| cut -d= -f2)` to bypass hostname validation and pin by fingerprint instead |
| `pvecm add` DNS | The new node must be able to resolve the target cluster node's hostname before running `pvecm add` (TLS cert hostname check). | Add all cluster node hostnames to `/etc/hosts` on the new node before running `pvecm add` |
| `pvecm add` TTY | `pvecm add` reads the root password via TTY. In non-interactive SSH sessions (e.g., scripted provisioning), the password prompt hangs. | Pipe the password via stdin: `echo "$PASS" \| pvecm add <host> --fingerprint <fp> --force` |
| New PVE node networking | A freshly installed PVE node may have its default gateway set to the network address (e.g., `10.0.0.0`) instead of the actual router IP -- caused by installer auto-detection errors. | Fix `/etc/network/interfaces` to set `gateway 10.0.0.1` (or the correct router IP), then run `ip route replace default via <gateway>` before running `apt` or any network-dependent command |
| VFIO / GPU passthrough | `echo "VID DID" > /sys/bus/pci/drivers/vfio-pci/new_id` claims unbound PCIe devices dynamically at runtime. The device must already be unbound from its native driver (e.g., `nvidia`, `nouveau`) before the `new_id` write will bind it to `vfio-pci`. | Unbind the driver first: `echo <BDF> > /sys/bus/pci/drivers/<driver>/unbind`, then write to `new_id`. For permanent binding, add the device IDs to `/etc/modprobe.d/vfio.conf` (`options vfio-pci ids=<vid>:<did>`) and regenerate initramfs |
| VFIO passthrough crash | `rombar=1` (the Proxmox default) causes QEMU to call `pci_map_rom()`, which attempts to map the GPU ROM BAR into the VGA legacy region `0xc0000-0xdffff`. If the host PCI bus window only covers a subset of that range (e.g., `0xc4000-0xc7fff`), a resource conflict triggers a host kernel panic within seconds of VM start. EFI framebuffer (`simplefb`/`sysfb`) holding GPU BARs as `BOOTFB` reservations and VGA arbitration interference compound the issue. | Set `rombar=0` in the hostpci config; add `initcall_blacklist=sysfb_init` to `GRUB_CMDLINE_LINUX_DEFAULT`; add `options vfio-pci ids=<vid>:<did> disable_vga=1` to `/etc/modprobe.d/vfio.conf`. See deployment-troubleshooting.md for full diagnosis |
| VFIO `rombar=0` and NVIDIA driver | The NVIDIA Linux driver does NOT need the PCI ROM BAR -- it loads firmware internally. `rombar=0` is safe and correct for all compute-only (non-display) GPU passthrough. Setting `rombar=1` for CUDA-only workloads provides no benefit and risks the `pci_map_rom` crash described above. | Use `rombar=0` for any GPU passed through exclusively for compute (no display output from the guest) |
| Terraform VFIO | The bpg/proxmox `hostpci` block `rombar` attribute defaults to `true` in Terraform modules. For compute-only GPU passthrough, this is the wrong default and will cause the `pci_map_rom` crash on some hardware. | Always set `rombar = false` (or the module-level `gpu_rombar = false` if using a wrapper variable) for CUDA-only GPU VMs |

## GPU / NVIDIA Passthrough

| Area | Gotcha | Fix |
|------|--------|-----|
| VFIO rombar | `rombar=1` (Proxmox default) causes QEMU to call `pci_map_rom()` which tries to map the GPU ROM BAR into the VGA legacy region `0x000c0000-0x000dffff`. If the host PCI bus window does not cover that range (e.g., `0x000c4000-0x000c7fff` on X299), the resource conflict triggers a host kernel panic. | Set `rombar=0` on every `hostpci` line for compute-only (non-display) GPUs. The NVIDIA Linux driver does not need the PCI ROM -- it loads firmware internally. |
| VFIO rombar | NVIDIA Linux driver does NOT need the PCI ROM for compute workloads | `rombar=0` is correct for all CUDA-only GPU passthrough; do not set `rombar=1` for non-display GPUs |
| VFIO EFI framebuffer | EFI stub initialises a framebuffer using the GPU's BARs and marks them as `BOOTFB` reservations. IOMMU/VFIO DMA mapping for those BARs is then blocked, causing VM start to hang or crash. | Add `initcall_blacklist=sysfb_init` to `GRUB_CMDLINE_LINUX_DEFAULT` on the Proxmox host and rebuild initramfs |
| VFIO VGA arbitration | Without `disable_vga=1` in the vfio-pci module options, the kernel VGA arbitration code may hold the VGA region open on behalf of vfio-pci, interfering with device DMA setup. | Add `options vfio-pci ids=<vid>:<did> disable_vga=1` to `/etc/modprobe.d/vfio.conf` and rebuild initramfs |
| VFIO audio FLR | The HDMI audio function (`xx:00.1`) on most NVIDIA GPUs has no FLR capability (no `reset_method` sysfs entry). VFIO falls back to group-level bus reset for the whole IOMMU group. On X299/Skylake-X, this group bus reset crashes the host. | Restrict the GPU function to FLR-only: `echo flr > /sys/bus/pci/devices/0000:<BDF>/reset_method`. Persist via udev: `ACTION=="bind", SUBSYSTEM=="pci", ATTR{vendor}=="0x10de", ATTR{device}=="<vid>", ATTR{reset_method}="flr"` in `/etc/udev/rules.d/99-vfio-flr.rules` |
| VFIO vendor-reset | The `vendor-reset` kernel module supports AMD GPU reset only -- it has no implementation for NVIDIA GPUs | Building or loading `vendor-reset` for NVIDIA passthrough provides no benefit and taints the kernel; ignore suggestions to use it with NVIDIA |
| VFIO KVM vs LXC | RTX 3060 (GA106/Ampere) VFIO passthrough to QEMU on X299/Skylake-X is not reliable. After fixing rombar, sysfb, VGA arb, and FLR, the host kernel still faults during guest NVIDIA driver initialisation (~3-4 min into VM boot). This is a known incompatibility with Ampere DMA init + X299 PCIe handling. Hardware works correctly with native NVIDIA drivers. | Use LXC + host NVIDIA driver instead: install NVIDIA drivers on the Proxmox host, create a privileged LXC container, and bind-mount `/dev/nvidia*` device nodes into it via `lxc.cgroup2.devices.allow` + `lxc.mount.entry`. No VFIO or IOMMU configuration needed. |
| LXC + NVIDIA | The bpg/proxmox Terraform provider does not expose `lxc.cgroup2.devices.allow` or `lxc.mount.entry` LXC config fields. These are raw LXC kernel interface entries that the provider does not model. | Write them directly to `/etc/pve/lxc/<vmid>.conf` via SSH using a `null_resource` provisioner after container creation. The Proxmox cluster filesystem is writable from any cluster node. |
| LXC + NVIDIA | Unprivileged LXC containers remap UIDs (user namespace isolation). `cgroup2.devices.allow` rules are evaluated against the remapped UID, breaking device access for NVIDIA devices that expect root UID. | Use `unprivileged = false` for LXC containers that require NVIDIA device bind-mounts. Privileged containers share the host UID namespace and honour device allow rules correctly. |
| LXC + NVIDIA | NVIDIA device nodes require specific cgroup v2 character device allow entries. Missing any one prevents `nvidia-smi` or CUDA from working inside the container. | Allow: `c 195:* rwm` (`/dev/nvidia0`, `/dev/nvidiactl`), `c 236:* rwm` (`/dev/nvidia-uvm`, `/dev/nvidia-uvm-tools`), `c 10:235 rwm` (`/dev/nvidia-modeset`). Also bind-mount all five device nodes with `none bind,optional,create=file`. Note: major number 236 applies to older drivers; see below for NVIDIA 580+ |
| LXC + NVIDIA | `optional,create=file` mount flags prevent container start failure when the host NVIDIA driver is not loaded (e.g., after driver reinstall or host maintenance). Without `optional`, a missing `/dev/nvidia0` causes the container to refuse to start. | Always include `optional,create=file` on all `lxc.mount.entry` lines for NVIDIA device nodes. |
| LXC + NVIDIA | In NVIDIA driver 580+, the `nvidia-uvm` major device number is dynamically assigned by the kernel (e.g., `510` on kernel 6.17), NOT the previously hardcoded `236`. Using `c 236:* rwm` in `lxc.cgroup2.devices.allow` will grant access to the wrong device class and CUDA will fail inside the container. | Read the actual major number at runtime: `cat /sys/module/nvidia_uvm/parameters/uvm_dev_major`. Use `c *:* rwm` for broad allow (simplest for a trusted privileged container), or use `c 510:* rwm` if the major is confirmed stable on the host kernel. Update the Terraform null_resource provisioner to detect the major dynamically. |
| LXC + NVIDIA | In NVIDIA 580+, `/dev/nvidia-modeset` does NOT exist as a standalone character device node. Modeset functionality is accessed via `ioctl` on `/dev/nvidiactl`. The `c 10:235 rwm` cgroup allow entry and the `lxc.mount.entry` bind-mount for `nvidia-modeset` are no longer needed (and will fail silently since the device node does not exist). | Remove the `nvidia-modeset` bind-mount entry. Keep `c 195:* rwm` (nvidia0/nvidiactl) and the dynamic uvm major entry. Verify with `ls -la /dev/nvidia*` on the host after driver install. |
| LXC + NVIDIA | `nvidia-uvm` device nodes (`/dev/nvidia-uvm`, `/dev/nvidia-uvm-tools`) are created lazily -- they do not exist until the first userspace access (e.g., `nvidia-smi` or any CUDA call). If the LXC container starts before `nvidia-uvm` has been initialised on the host, the bind-mount entries will fail silently (with `optional`) and CUDA will fail inside the container. | Create a systemd oneshot service on the Proxmox host that runs `nvidia-smi` before any LXC containers start. This forces uvm device node creation. Example: `/etc/systemd/system/nvidia-uvm-init.service` with `ExecStart=/usr/bin/nvidia-smi` and `Before=lxc.service`. |
| LXC + NVIDIA | The `vendor-reset` DKMS kernel module (commonly suggested for AMD GPU reset) supports AMD GPUs ONLY. Loading it on a system with NVIDIA GPUs provides no benefit and causes spontaneous host crashes (kernel panics unrelated to GPU operations). | Remove completely: `dkms remove vendor-reset/0.1.1 --all && apt-get remove --purge dkms-vendor-reset`. Do not install on any host running NVIDIA drivers. |
| LXC + pct exec | `pct exec <vmid> -- apt-get install ...` on kernel 6.17 + X299/Skylake-X + cgroup2 with a privileged LXC container causes repeatable host crashes (kernel panic or machine check). The crash occurs during apt workloads inside the container but never during idle operation. Root cause is unknown (suspected cgroup2 + X299 PCIe + kernel 6.17 interaction). BIOS update (P1.70 to P2.50 on ASRock X299 Taichi) did NOT resolve the pattern. | Avoid `pct exec` for package installation on affected hardware. SSH directly into the container instead (`ssh root@<container-ip>`), or run Docker directly on the PVE host without a container intermediary. |
| Terraform comfyui | `gpu_rombar` variable in the `comfyui` VM module defaults to `true`, which is wrong for compute GPUs | Always set `gpu_rombar = false` when enabling GPU passthrough to a CUDA-only (non-display) VM |

## K3s / Kubernetes

| Area | Gotcha | Fix |
|------|--------|-----|
| K3s bootstrap | `--cluster-init` is for embedded etcd only, not external PostgreSQL datastores | With external PostgreSQL, all servers use `--datastore-endpoint` + `--token` |
| K3s bootstrap | Initial bootstrap token must be a simple string (not `K10...` format) | `K10` format is for post-bootstrap joins only |
| K3s health check | `/readyz` endpoint returns 401 for unauthenticated requests -- NOT a sign that the server is unhealthy | Health check scripts and agent probes must treat 200, 401, and 403 as "reachable" (only connection refused or timeout means down) |
| K3s topology | Agent nodes need `--server` flag to join | Servers discover each other via PostgreSQL |
| K3s registry mirror `/v2` path | containerd generates `override_path = true` for path-based endpoints, which causes it to append the image path WITHOUT the `/v2/` prefix. For a Nexus Docker proxy, this constructs an invalid URL and Nexus returns `400 Not a docker request`. | Append `/v2` to each endpoint URL in `registries.yaml` (e.g., `http://10.0.0.202:8081/repository/docker-hub/v2`). The template at `ansible/templates/k3s-registries.yaml.j2` already does this -- do not remove the `/v2` suffix. |
| K8s DNS | Pods cannot resolve Tailscale MagicDNS (`.ts.net`) -- CoreDNS uses cluster DNS (10.43.0.10), not host resolver | Use LAN IPs for in-cluster access to Tailscale-only services |
| CoreDNS `hosts` plugin | The CoreDNS `hosts` plugin can only appear **once** per server block in the Corefile. Adding a second `hosts {}` stanza (e.g., in a ConfigMap override) causes CoreDNS to refuse to start or silently ignore the second entry. | Use `rewrite name exact <src> <dst>` to redirect specific hostnames to existing records, instead of adding a second `hosts` block. Multiple `rewrite` directives in the same server block are allowed. |
| CoreDNS ConfigMap ordering | A CoreDNS override ConfigMap that adds custom DNS entries (e.g., to resolve an OIDC issuer hostname for OAuth2 Proxy) must be applied before the dependent service starts. If the ConfigMap lives in `platform-configs` but OAuth2 Proxy is in `platform-controllers`, the Proxy may start (and fail OIDC discovery) before the DNS override is live. | Place CoreDNS override ConfigMaps in `platform-controllers` alongside CoreDNS itself, not in `platform-configs`. This ensures the DNS override is applied before any downstream service that depends on custom name resolution. |
| K8s StatefulSet | `volumeClaimTemplates` are immutable | Delete Helm release secrets (`sh.helm.release.v1.<name>.v*`) to force fresh install |
| K8s StatefulSet | Helm chart version downgrade followed by re-upgrade can trigger immutable field errors on StatefulSets even when the final desired state matches | The downgrade may have altered `volumeClaimTemplates` or other immutable fields in the intermediate state; delete the StatefulSet (not the PVCs) and let Helm recreate it, or use `spec.upgrade.force: true` in the Flux HelmRelease |
| K8s probes | `kubectl apply` cannot change a probe's handler type in-place (e.g., `httpGet` to `tcpSocket`) -- returns `may not specify more than 1 handler type` | Delete the Deployment and recreate it; or use Flux `spec.upgrade.force: true` to handle the recreation automatically |
| K8s Secrets | `secretKeyRef` without `optional: true` blocks pod startup (`CreateContainerConfigError`) if the secret does not exist yet | Add `optional: true` or ensure secret is created first |
| K8s Secrets | When many `secretKeyRef` env vars come from a credential Secret populated post-deploy (e.g., Homepage widgets), ALL need `optional: true` -- one missing blocks the entire pod | Pattern: set `optional: true` on every `secretKeyRef` for late-binding credentials |
| K8s PV | PersistentVolume `spec` fields (including `nfs.path`) are immutable after creation | Delete PV + PVC and let Flux recreate; NFS data is unaffected by PV deletion |
| K8s networking | Pod traffic to same-LAN PostgreSQL routes via node IP (10.0.0.x), not pod CIDR (10.42.0.0/16) | Add `10.0.0.0/24` to `pg_hba.conf` |
| Flux | Kustomize values patches must be strategic merge patches (wrap in HelmRelease resource), not raw Helm values | Wrap patch in full HelmRelease resource structure |
| Flux | `spec.upgrade.force: true` handles StatefulSet immutable field changes | Deletes + recreates the Helm release |
| Flux | `flux bootstrap github` writes an SSH URL (`ssh://git@github.com/...`) into `flux-system/gotk-sync.yaml`, but if the GitRepository was previously configured with an HTTPS URL, Flux fails with authentication errors | Ensure the GitRepository URL scheme matches the bootstrap method; delete and re-bootstrap if switching between SSH and HTTPS |
| Flux | Kustomization circular dependency: if monitoring-configs depends on monitoring-controllers, ExternalSecrets in configs cannot create Secrets needed by Deployments in controllers | Co-locate ExternalSecret with its consumer |
| Flux | All platform HelmReleases need `install.remediation.retries: 3` | Handles dependency chain chicken-and-egg (controllers -> configs ordering) |
| ingress-nginx | `configuration-snippet` annotation is blocked by default in ingress-nginx 1.9+ (`allow-snippet-annotations=false`). Using it returns an admission webhook rejection. | Use `nginx.ingress.kubernetes.io/proxy-set-headers` pointing to a ConfigMap instead. ConfigMap keys are header names; values support nginx variables (e.g., `$http_upgrade`). Annotation format: `<namespace>/<configmap-name>`. |
| ingress-nginx | Keycloak JWTs with `realm-management` roles attached are large (>4 KB); the default nginx proxy buffer size (4k/8k) causes upstream responses to be written to temp files and silently dropped, resulting in 502 errors on the OAuth2 Proxy callback. | Add annotation `nginx.ingress.kubernetes.io/proxy-buffer-size: "128k"` to the OAuth2 Proxy `Ingress` resource. |
| Flux | Migrating `HelmRepository` URLs to a new proxy (e.g., Nexus) before the proxy repos exist causes 404s on all HelmRepositories and cascade-fails all dependent Kustomizations via the health-check dependency chain | Create the proxy repos BEFORE merging the manifest changes; test proxy reachability from a feature branch first |
| Flux | Flux Kustomization reconciliation is ATOMIC -- one failing resource blocks ALL resources in that Kustomization from being applied | Split large Kustomizations into smaller units, or isolate problematic resources; a single invalid manifest prevents the entire Kustomization from reconciling |
| Flux | `kustomize build` dry-run fails for dynamically provisioned PVCs that are already bound to PVs in the cluster but lack `volumeName` in the manifest | Add `volumeName: <pv-name>` to the PVC manifest once bound, matching the actual PV; without this, Flux dry-run creates a hypothetical PVC that conflicts with the existing bound PVC |
| K8s PV | PersistentVolume `spec.nfs.path` (and most other `spec` fields) are immutable after creation | Delete the PV and PVC, then let Flux recreate them; NFS data on the server is unaffected by PV/PVC deletion |
| K8s PV/PVC | Protection finalizers (`kubernetes.io/pv-protection`, `kubernetes.io/pvc-protection`) can deadlock PV/PVC deletion -- PVC waits for pod detachment, PV waits for PVC deletion | Patch both resources to remove finalizers: `kubectl patch pv <name> -p '{"metadata":{"finalizers":null}}'` and same for PVC; then delete normally |

## Cloud-init

| Area | Gotcha | Fix |
|------|--------|-----|
| cloud-init | `write_files` with `owner: <custom_user>` fails -- runs in init-network stage before user creation | Add `defer: true` to move to final stage |
| cloud-init | `lock_passwd: false` without `passwd:` does NOT allow console login on Ubuntu 24.04 | PAM rejects empty passwords; add a `chpasswd:` block |
| VLAN | If PVE hosts are on VLAN 2 untagged via vmbr0, do NOT set `vlan_id` on VMs | Causes double-tagging and network isolation |
| QEMU agent | Not running = `terraform plan` hangs for minutes (timeout per VM) | Start agent in cloud-init runcmd; reduce timeout from 15m to 2m |
| Interface naming | Proxmox VMs with virtio NICs on Ubuntu 24.04 cloud images use `eth0`, NOT `ens18` -- predictable naming (`ens*`) requires PCI slot metadata absent in cloud images | Use `eth0` in keepalived config and network scripts; verify with `ip link show` |

## Scripting / Templating

| Area | Gotcha | Fix |
|------|--------|-----|
| Jinja2 + Bash | `${#array[@]}` contains `{#` which Jinja2 interprets as a comment start (`{# ... #}`), silently swallowing the rest of the template output | Wrap Bash-native syntax in `{% raw %}...{% endraw %}` blocks inside Jinja2 templates (e.g., Ansible `.j2` files) |
| Terraform templatefile | Bash array length syntax `${#ERRORS[@]}` inside `.tftpl` files is parsed as an invalid Terraform template expression -- `terraform plan` fails with "invalid escape sequence" | Escape with `$${#ERRORS[@]}` -- the double `$$` produces a literal `$` in the output; this is distinct from the Jinja2 problem (Terraform uses `${...}` interpolation natively) |
| Bash arithmetic | `((var++))` returns exit code 1 when `var` is 0 (because 0 is falsy in arithmetic context); with `set -euo pipefail`, this kills the script | Avoid `(( ))` for increment/decrement of potentially-zero variables; use `TOTAL_FAIL=${#ERRORS[@]}; TOTAL_OK=$(( N - TOTAL_FAIL ))` instead, or append `\|\| true` after the arithmetic expression |
| yq + snap confinement | `yq` installed via snap cannot read files from `/tmp/` -- snap's filesystem confinement sandbox blocks access to paths outside standard home/data dirs. `yq eval '...' "${f}"` fails silently or with a permission error when `${f}` is a temp file. | Use stdin redirect: `yq eval '...' - < "${f}"`. The shell opens the file and passes the fd; yq only sees stdin and never needs path access. Applies to any snap-confined binary. |
| ShellCheck SC2002 | "Useless cat" -- `cat file \| cmd` should be `cmd < file`. The CI shellcheck (`koalaman/shellcheck-alpine:stable`) may be stricter than a locally installed version, causing CI-only failures on code that passes locally. | Always run `shellcheck -x` locally before pushing. Use `yq eval '...' - < "${f}"` not `cat "${f}" \| yq eval '...' -`. The stdin redirect form avoids both SC2002 and snap confinement. |
| curl API key security | `-H "X-Api-Key: ${TOKEN}"` puts the token value in the process argv -- visible to other users via `/proc/<pid>/cmdline` and `ps aux`. This is a security concern on multi-user systems. | Write headers to a curl config file and use `--config file`: `printf 'header = "X-Api-Key: %s"\n' "${TOKEN}" > "${CONFIG_FILE}"`. Also write the JSON request body to a temp file and pass `--data-binary @file` (avoids shell escaping issues with large base64 strings). |
| trap EXIT temp file cleanup | Registering `trap cleanup EXIT` before `mktemp` is called means `cleanup()` may reference unset variables if the trap fires before the `mktemp` assignment. `rm -f ""` with `set -u` can abort the script. | Initialize temp file variables to empty strings before the trap, then call `mktemp` after: `VAR1="" VAR2=""; cleanup() { rm -f ${VAR1:+"${VAR1}"} ${VAR2:+"${VAR2}"}; }; trap cleanup EXIT; VAR1=$(mktemp); VAR2=$(mktemp)`. ## GitLab Runner (K3s Shell Executor)

| Area | Gotcha | Fix |
|------|--------|-----|
| `glrt-` registration tokens (GitLab 16+) | Runner attributes like `--tag-list`, `--run-untagged`, `--locked`, and `--access-level` are configured server-side when creating the registration token in the GitLab UI. They CANNOT be passed as flags to `gitlab-runner register` -- the flags are silently ignored when using `glrt-` tokens. | Set all runner attributes in the GitLab UI token creation form. The `register` command only needs `--url`, `--token`, `--name`, and `--executor`. |
| GitLab instance URL in K3s runner | K3s VMs have no Tailscale. Using a Tailscale hostname (`.ts.net`) as `gitlab_instance_url` in the runner registration causes `x509: unknown authority` or connection failure because the VM cannot resolve the name. | Use the LAN nip.io ingress URL (e.g., `https://gitlab.10.0.0.201.nip.io`) for `gitlab_instance_url` in the Ansible role. The nip.io hostname resolves to the MetalLB VIP and is reachable from K3s VMs without Tailscale. |
| CA certificate PEM validity | A CA certificate written to `/usr/local/share/ca-certificates/` must be a valid PEM file. A single typo in the base64 body (e.g., `OULFH6R` instead of `OUFH6R`) causes `openssl x509` to reject it and `update-ca-certificates` to fail silently. `gitlab-runner` then fails to connect with `x509: certificate signed by unknown authority` even though the cert file exists. | Verify the PEM file with `openssl x509 -noout -text -in <cert>.crt` before deploying. Base64 errors surface as `unable to load certificate: ... no start line`. |
| `gitlab-runner list` output | `gitlab-runner list` writes its output to STDERR, not stdout. Capturing with `$(gitlab-runner list)` returns an empty string. | Use `2>&1` redirect: `runner_list=$(gitlab-runner list 2>&1)`. Check for runner presence with: `(runner_name + ' ') not in runner_list` -- the runner name in list output is always followed by spaces before the Executor column; use the trailing space to avoid substring false-positives (e.g., `myrunner ` vs `myrunner-2`). |
| Ansible runner registration idempotency | Checking `runner_name in runner_list_output.stderr` is fragile if multiple runners share a common name prefix -- `myrunner` matches `myrunner-2`. | Append a space to the check: `(runner_name + ' ') not in runner_list_output.stderr`. The space is always present because `gitlab-runner list` pads runner names before the Executor column. |

## GitHub Actions / CI

| Area | Gotcha | Fix |
|------|--------|-----|
| GitHub Actions | `kubectl --field-selector=status.successful=1` is unreliable for filtering Jobs by integer status fields -- the field selector spec only guarantees support for `metadata.name` and `metadata.namespace` on most resource types; additional fields may silently return all or no resources depending on the Kubernetes version | Filter via `jq` instead: `kubectl get jobs -n <ns> -o json \| jq '[.items[] \| select(.status.succeeded > 0)]'` |
| GitHub Actions | `grep -oP` (Perl-compatible regex) is not reliably available on `ubuntu-latest` GitHub Actions runners -- GNU grep build may not include PCRE support | Use `sed` for version extraction: `sed -n 's/.*\([0-9]\+\.[0-9]\+\.[0-9]\+\).*/\1/p'` or `python3 -c "import re; ..."` for complex patterns (Python 3 always available on GitHub-hosted runners) |
| GitHub Actions | Tailscale-only homelabs cannot receive inbound webhook calls from GitHub -- webhook-based CI exporters and inbound HTTP patterns do not work | Use outbound-only approaches: `tailscale/github-action@v3` with a reusable ephemeral auth key (`TS_AUTH_KEY`) lets the runner join the tailnet and reach cluster resources; ephemeral nodes auto-expire after the job ends |
| GitHub Actions | `kubectl apply` cannot change a probe's handler type in-place (e.g., `httpGet` to `tcpSocket`) -- returns `may not specify more than 1 handler type` | Delete the Deployment and recreate it, or use Flux `spec.upgrade.force: true` to handle the recreation automatically |
| GitHub SSH | GitHub port 22 may be blocked on some networks (corporate firewalls, restrictive ISPs). Flux `GitRepository` sources using `ssh://git@github.com/...` fail with connection timeout. | Use the SSH-over-HTTPS endpoint: `ssh://git@ssh.github.com:443/<org>/<repo>.git`.
